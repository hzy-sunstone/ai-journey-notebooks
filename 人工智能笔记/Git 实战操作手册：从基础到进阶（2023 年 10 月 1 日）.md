## 二、日常开发核心流程（高频使用）

### 1. 仓库初始化与克隆

```Bash

# 新建本地仓库
git init

# 克隆远程仓库（推荐HTTPS方式）
git clone https://github.com/用户名/仓库名.git

# 克隆指定分支
git clone -b 分支名 https://github.com/用户名/仓库名.git
```

### 2. 核心工作流（add → commit → pull → push）

```Bash

# 查看文件状态（红=未暂存，绿=已暂存）
git status

# 暂存所有修改文件
git add .

# 暂存指定文件
git add 文件名1 文件名2

# 提交暂存区到本地仓库（规范格式：类型: 描述）
git commit -m "feat: 新增用户登录功能"

# 拉取远程最新代码（推荐使用rebase避免冗余合并）
git pull --rebase origin main

# 推送本地代码到远程（首次推送需绑定上游分支）
git push -u origin main

# 后续推送（已绑定上游分支）
git push
```

## 三、标签（Tag）管理详解（版本发布必备）

### 1. 标签类型与创建

|标签类型|命令|适用场景|
|---|---|---|
|轻量标签|`git tag v1.0.0`|临时标记，无说明信息|
|附注标签|`git tag -a v1.0.0 -m "I.R.I.S-Agent V1.0.0 正式发布"`|正式版本发布，包含详细说明|
|签名标签|`git tag -s v1.0.0 -m "签名发布版本"`|安全性要求高的场景，需GPG密钥|
### 2. 关键标签操作

```Bash

# 查看所有标签
git tag

# 查看标签详细信息
git show v1.0.0

# 给历史提交打标签（指定提交哈希）
git tag -a v1.0.0 提交哈希 -m "描述信息"

# 推送单个标签到远程
git push origin v1.0.0

# 推送所有标签到远程（谨慎使用）
git push origin --tags

# 删除本地标签
git tag -d v1.0.0

# 删除远程标签（两种方式）
git push origin --delete v1.0.0
git push origin :refs/tags/v1.0.0
```

### 3. 标签最佳实践

- **版本号规范**：遵循`主版本.次版本.修订版本`（如v1.0.0）

- **发布必打标签**：每次正式发布都应创建附注标签并添加详细说明

- **合并提交处理**：合并后建议打标签标记重要版本，替代默认的"This is a combination of 2 commits"

- **标签不可随意修改**：已推送的标签应视为只读，如需修改应创建新标签

## 四、分支管理（多人协作必备）

### 1. 分支基础操作

```Bash

# 查看所有本地分支（*表示当前分支）
git branch

# 查看所有远程分支
git branch -r

# 创建新分支
git branch 分支名

# 切换分支
git checkout 分支名

# 创建并切换分支（简写）
git checkout -b 分支名

# 合并指定分支到当前分支
git merge 分支名

# 删除本地分支（已合并）
git branch -d 分支名

# 强制删除本地分支（未合并）
git branch -D 分支名

# 删除远程分支
git push origin --delete 分支名
```

### 2. 分支命名规范（推荐）

```Plain Text

feature/功能名  # 新功能开发（如feature/user-auth）
bugfix/问题名   # 缺陷修复（如bugfix/login-error）
release/版本号  # 发布分支（如release/v1.0）
hotfix/紧急修复 # 线上紧急修复（如hotfix/critical-security）
```

## 五、提交历史管理（清理混乱历史）

### 1. 查看提交历史

```Bash

# 查看简洁提交历史
git log --oneline

# 查看详细提交历史（包含分支合并图）
git log --graph --decorate --oneline

# 查看指定作者的提交
git log --author="你的名字"
```

### 2. 提交历史修改（个人分支适用）

```Bash

# 修改最近一次提交信息
git commit --amend -m "新的提交信息"

# 合并多个提交（推荐使用交互式rebase）
git rebase -i HEAD~3  # 合并最近3次提交

# 软重置（删除提交记录，保留工作区和暂存区）【最安全】
git reset --soft 目标提交哈希 

# 混合重置（默认，删除提交记录，保留工作区，清空暂存区）
git reset 目标提交哈希 

# 硬重置（删除提交记录和工作区修改，谨慎使用！）
git reset --hard 目标提交哈希 
```

### 3. 强制推送（仅个人仓库/分支使用）

```Bash

# 安全强制推送（避免覆盖他人代码）
git push --force-with-lease origin main

# 普通强制推送（风险较高）
git push -f origin main
```

## 六、撤销与恢复操作（应急必备）

### 1. 撤销工作区修改（未add）

```Bash

# 撤销指定文件
git checkout -- 文件名

# 撤销所有工作区修改
git checkout .
```

### 2. 撤销暂存区修改（已add未commit）

```Bash

# 撤销所有暂存
git reset

# 撤销指定文件暂存
git reset 文件名
```

### 3. 恢复误删的提交/分支

```Bash

# 查看所有操作记录（包含已删除的提交）
git reflog

# 恢复到指定操作（通过reflog找到哈希）
git reset --hard 操作哈希
```

## 七、冲突处理（团队协作常见）

1. **冲突出现标志**：文件中出现`<<<<<<< HEAD`、`=======`、`>>>>>>> 分支名`标记

2. **解决步骤**：

    ```Bash
    
    # 1. 打开冲突文件，手动编辑保留需要的代码
    # 2. 标记冲突已解决
    git add 冲突文件名
    # 3. 继续rebase（如果是pull --rebase产生的冲突）
    git rebase --continue
    # 4. 若无法解决，可终止rebase
    git rebase --abort
    ```

## 八、关键注意事项与避坑指南

### 绝对禁止的操作

1. ❌ 在公共分支（main/master）使用`git push -f`强制推送

2. ❌ 使用`git reset --hard`前未备份重要代码

3. ❌ 提交包含敏感信息（密码、密钥、API密钥）的文件

4. ❌ 在未提交的情况下执行`git pull`导致冲突覆盖修改

### 最佳实践

1. ✅ 提交前先`git pull --rebase`，减少冲突概率 

2. ✅ 小步提交：每个提交只包含一个完整功能/修复，便于回滚

3. ✅ 写清晰的提交信息，遵循规范格式：`类型: 描述`（如feat、fix、docs、refactor）

4. ✅ 个人分支可自由整理历史，公共分支保持线性干净

5. ✅ 发布版本必须打标签，便于追溯和回滚

## 十、常见问题解决方案

### 1. 合并后提交信息问题

**问**：合并后显示"This is a combination of 2 commits"，需要写tag吗？  

**答**：建议创建标签标记重要版本，同时修改提交信息为有意义的内容：

```Bash

# 合并后修改提交信息
git commit --amend -m "feat: 合并用户模块与支付模块，发布v1.0.0"
# 打标签
git tag -a v1.0.0 -m "版本说明"
# 推送标签
git push origin v1.0.0
```

### 2. 找不到VS Code菜单选项

- 更新VS Code到最新版本

- 确保选中的提交不是当前分支的最新提交（重置选项只在非最新提交显示）

- 使用命令面板（Ctrl+Shift+P）搜索"Git: Reset" 

### 3. 远程历史混乱的终极解决方案

```Bash

# 1. 备份本地代码
# 2. 拉取远程最新代码
git fetch origin
# 3. 强制重置本地分支与远程一致
git reset --hard origin/main
# 4. 重新添加修改并提交
git add .
git commit -m "feat: 干净版本发布"
git push
```
